# cpk-ui 0.7: make component behavior visible

A component library becomes difficult to trust when its source looks correct
but its states are hard to inspect. cpk-ui 0.7 treats examples, tests, and web
rendering as one release surface.

## Why kstyled remains the styling layer

React Native components need native press, focus, animation, and accessibility
behavior. A web-first abstraction does not remove that platform work. kstyled
0.4 keeps the API familiar while moving static declarations to build time and
making unsupported transforms fail in CI.

That gives cpk-ui one styling contract without forcing the component layer into
a DOM-shaped API.

## Storybook as an engineering tool

The new Storybook is organized by component purpose rather than source folder.
Every public component has realistic states, responsive web coverage, controls,
and accessibility inspection. Examples use controlled data and product-shaped
copy so clicking a control produces an observable result.

The workbench is useful to maintainers because it exposes regressions quickly,
and useful to consumers because it shows the contract they can actually rely
on.

## Performance work that changes shipped bytes

The previous icon path loaded a 6.4 MB IcoMoon selection file to reconstruct a
glyph map already represented by the typed icon list. Version 0.7 derives the
map from that list and excludes the selection file from npm.

The resulting dry-run is about 3.9 MB compressed and 6.8 MB unpacked, down from
5.5 MB and 13.1 MB. Fonts remain the largest intentional package cost.

## Release discipline

The publish workflow now requires component and story coverage, TypeScript,
Jest, the package build, E2E smoke tests, a static Storybook build, and tarball
inspection. npm publishing remains restricted to GitHub Actions on main and
includes provenance.

This is the practical direction for cpk-ui: preserve native behavior, make
cross-platform differences inspectable, and optimize costs that can be measured
in the package developers install.
